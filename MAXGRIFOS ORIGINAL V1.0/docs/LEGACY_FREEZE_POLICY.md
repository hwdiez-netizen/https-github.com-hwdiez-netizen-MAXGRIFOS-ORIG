# LEGACY FREEZE POLICY — MAXGRIFOS ERP V2

## Estado oficial

LEGACY_FREEZE.

El código legacy queda congelado como referencia de solo lectura, consulta técnica y auditoría forense.

Legacy NO gobierna el nuevo proyecto.

Legacy NO define layout.

Legacy NO define navegación.

Legacy NO define bottom navigation bar.

Legacy NO define AppShell.

Legacy NO define NIS.

Legacy NO define UI/UX mobile-first.

Legacy NO define frontend objetivo.

Legacy NO puede reintroducir patrones visuales obsoletos.

Legacy NO puede imponerse sobre la navegación premium.

Legacy NO puede imponerse sobre la Constitución MAXGRIFOS ERP V2.

Legacy NO puede imponerse sobre el Navigation Interaction System.

Legacy NO puede imponerse sobre mobile-first real.

Legacy NO puede imponerse sobre contratos, handlers, idempotencia ni arquitectura V2.

---

## Rol permitido de Legacy

Legacy solo puede ser auditado como fuente de evidencia funcional, técnica o arquitectónica.

Legacy puede servir como referencia únicamente para extraer:

- comportamientos robustos;
- reglas de negocio útiles;
- contratos válidos;
- handlers válidos;
- flujos funcionales probados;
- estructuras lógicas que hayan demostrado utilidad;
- evidencia histórica de cómo resolvía un problema el sistema anterior.

Toda extracción desde Legacy debe ser selectiva, justificada y adaptada al nuevo marco MAXGRIFOS ERP V2.

Legacy puede orientar lógica funcional o arquitectura interna SOLO SI pasa filtro constitucional.

Legacy NO es una fuente automática de verdad.

Legacy NO se porta completo por defecto.

Legacy NO se copia por conveniencia.

Legacy NO se usa para acelerar a costa de deuda técnica.

---

## Filtro constitucional obligatorio

Antes de portar cualquier lógica, flujo, patrón o estructura desde Legacy, debe verificarse que NO rompa:

- Constitución MAXGRIFOS ERP V2;
- contratos;
- handlers;
- idempotencia;
- arquitectura V2;
- frontend objetivo;
- UI/UX mobile-first;
- navegación premium;
- NIS;
- AppShell V2;
- consistencia visual;
- persistencia;
- trazabilidad;
- mantenibilidad;
- separación de responsabilidades;
- experiencia de usuario en smartphone 9:16.

Si una parte de Legacy rompe cualquiera de esos puntos, NO se porta.

Si una parte de Legacy sirve parcialmente, se extrae solo lo útil y se adapta de forma quirúrgica al marco V2.

Si Legacy contiene deuda técnica, esa deuda queda bloqueada.

Si adaptar Legacy crea nueva deuda técnica, esa adaptación queda bloqueada.

---

## Deuda técnica prohibida

Está prohibido copiar deuda técnica legacy.

Está prohibido arrastrar deuda técnica legacy.

Está prohibido crear nueva deuda técnica para adaptar Legacy.

Está prohibido usar Legacy como excusa para romper Constitución, NIS, UI/UX, AppShell, contratos, handlers o idempotencia.

Está prohibido usar Legacy como rector visual, arquitectónico o interactivo.

Está prohibido reintroducir navegación legacy como navegación dominante.

Está prohibido reintroducir layout legacy como layout dominante.

Está prohibido reintroducir patrones visuales legacy que contradigan mobile-first real.

---

## Regla de conflicto

Si existe conflicto entre Legacy y MAXGRIFOS ERP V2, domina MAXGRIFOS ERP V2.

Si existe conflicto entre Legacy y la Constitución MAXGRIFOS ERP V2, domina la Constitución.

Si existe conflicto entre Legacy y la navegación premium, domina la navegación premium.

Si existe conflicto entre Legacy y NIS, domina NIS.

Si existe conflicto entre Legacy y mobile-first real, domina mobile-first real.

Si existe conflicto entre Legacy y AppShell V2, domina AppShell V2.

Si existe conflicto entre Legacy y UI/UX, domina UI/UX mobile-first.

Si existe conflicto entre Legacy y frontend objetivo, domina el frontend objetivo V2.

Si existe conflicto entre Legacy y contratos, dominan los contratos.

Si existe conflicto entre Legacy y handlers, dominan los handlers.

Si existe conflicto entre Legacy y la idempotencia, domina la idempotencia.

---

## Regla de implementación

Toda tarea que use Legacy como referencia debe declarar explícitamente:

1. Qué parte de Legacy se auditó.
2. Qué comportamiento funcional se considera útil.
3. Qué evidencia justifica portarlo.
4. Qué deuda técnica se bloqueó.
5. Qué adaptación V2 se aplicó.
6. Qué contratos se respetaron.
7. Qué handlers se respetaron.
8. Cómo se preservó la idempotencia.
9. Cómo se protegió UI/UX mobile-first.
10. Cómo se evitó que Legacy dominara navegación, layout, AppShell o NIS.

Si una tarea no puede demostrar esos puntos, no puede declarar PASS.

---

## Frase rectora

MAXGRIFOS ERP V2 no es continuidad visual de Legacy.

Legacy está congelado.

Legacy es fuente de evidencia, no rector del nuevo sistema.

Legacy puede orientar lógica funcional solo si pasa filtro constitucional.

Legacy no gobierna UI/UX, navegación, NIS, AppShell ni frontend.

Lo útil se porta.

La deuda técnica se bloquea.

La navegación premium y NIS mobile-first dominan el nuevo proyecto.
